# NixOS Homelab Design Spec

## Overview

A reproducible, open-source NixOS homelab running on an Intel NUC. The entire system is declared in code, version-controlled, and deployable from a laptop. The repo contains zero secrets and can be cloned by anyone as a starting point for their own homelab.

## Hardware

- Intel NUC: i5 or i7, 16GB RAM, ~200GB SSD
- Connected to router via Ethernet with a static LAN IP

## Operating System

NixOS with Flakes. The flake pins nixpkgs and defines the NUC as a single host. The entire system config lives in this repo.

## Services

All services run as native NixOS modules (no Docker).

| Service | Purpose | Ports | NixOS Module | Caddy Virtual Host |
|---|---|---|---|---|
| AdGuard Home | DNS-level ad/malware blocking | 53 (DNS), 3000 (UI) | `services.adguardhome` | `adguard.home.lan` |
| Home Assistant | Home automation | 8123 | `services.home-assistant` | `hass.home.lan` |
| Caddy | Reverse proxy with auto-TLS | 80, 443 | `services.caddy` | — (is the proxy) |
| Tailscale | VPN mesh for remote access | — | `services.tailscale` | — (network layer) |

## Network Architecture

### Request Flow

```
Device DNS query → NUC:53 (AdGuard Home)
  → blocks ads/malware
  → forwards clean queries to upstream DNS (Cloudflare/Quad9)

Device HTTPS request → NUC:443 (Caddy)
  → adguard.home.lan  → AdGuard Home UI (:3000)
  → hass.home.lan     → Home Assistant UI (:8123)

Remote access (away from home)
  → Tailscale tunnel → same as above
```

### DNS Resolution for `.home.lan`

AdGuard Home has a DNS rewrite rule: `*.home.lan` resolves to the NUC's static LAN IP. Caddy then routes by hostname to the correct backend service. No per-device `/etc/hosts` editing needed.

### Router Configuration

The router's DNS setting is pointed to the NUC's static IP so all LAN devices automatically get ad-blocking and `.home.lan` resolution.

## Repository Structure

```

├── flake.nix                    # Entry point — pins nixpkgs, defines NUC host, wires modules
├── flake.lock                   # Pinned dependency versions (auto-generated)
├── hosts/
│   └── nuc/
│       ├── default.nix          # Machine config — imports all enabled modules
│       ├── hardware.nix         # Hardware-specific (generated by nixos-generate-config)
│       └── disk.nix             # Disk partitioning via disko
├── modules/
│   ├── adguard.nix              # AdGuard Home service + firewall rules
│   ├── home-assistant.nix       # Home Assistant service
│   ├── caddy.nix                # Caddy reverse proxy + virtual hosts
│   ├── tailscale.nix            # Tailscale VPN
│   └── common.nix               # Shared: locale, timezone, SSH, users, firewall base
├── tests/
│   ├── adguard-test.nix         # VM test: DNS queries + web UI
│   ├── caddy-test.nix           # VM test: proxy routing
│   └── integration-test.nix     # VM test: full stack boots, all services reachable
├── scripts/
│   └── setup.sh                 # Bootstrap: partitions disk, installs NixOS from this flake

│   └── workflows/
│       └── check.yml            # CI: flake check + VM tests
├── .gitignore
└── README.md                    # Step-by-step guide: bare NUC → running homelab
```

## Deployment

### Initial Setup (bare NUC → running homelab)

1. Flash NixOS minimal ISO to a USB stick
2. Boot the NUC from USB, connect via keyboard or SSH
3. Clone this repo onto the live environment
4. Run `scripts/setup.sh` which:
   - Partitions the disk using disko (declarative layout from `hosts/nuc/disk.nix`)
   - Prompts for secrets (Tailscale auth key, AdGuard admin password)
   - Writes secrets to `/etc/nixos/secrets/` on disk
   - Generates `hardware.nix` for the specific NUC hardware
   - Installs NixOS with the flake config
   - Reboots into the configured system
5. Post-boot: all services are running. Point router DNS to the NUC's static IP.

### Day-to-Day Changes (from laptop)

```bash
# Edit a module
vim modules/adguard.nix

# Validate
nix flake check

# Deploy to the NUC over SSH
./deploy.sh
```

`deploy.sh` wraps `colmena apply`, which builds the config and activates it on the NUC over SSH. Failed deployments can be rolled back with `nixos-rebuild switch --rollback` on the NUC.

## Secrets Management

### Principle

The repo contains zero secrets. It is safe to open-source and clone.

### What Needs Secrets

- Tailscale auth key (to join the user's tailnet)
- AdGuard Home admin password
- Home Assistant initial config / API tokens
- Any integration-specific credentials

### Approach

- Secrets are prompted interactively during `scripts/setup.sh` and written to `/etc/nixos/secrets/` on the NUC
- NixOS modules reference these paths for sensitive config
- The README documents what secrets are needed, their expected format, and where on disk they go
- `hosts/nuc/local.nix` (gitignored) holds user-specific overrides: static IP, hostname, domain name

### Gitignore

```
hosts/nuc/local.nix
secrets/
result
```

## Testing

### Level 1: Build Validation

```bash
nix flake check
```

Verifies all modules evaluate without errors. Catches typos, missing options, type mismatches. Fast — seconds, no VMs.

### Level 2: Per-Service VM Tests

Each test in `tests/` spins up a QEMU VM with the relevant modules and runs Python assertions:

- **adguard-test.nix**: Boots VM with AdGuard Home, asserts DNS on port 53 responds, web UI on 3000 returns HTTP 200
- **caddy-test.nix**: Boots VM with Caddy + a dummy backend, asserts proxy routing works by hostname
- **integration-test.nix**: Boots VM with full stack, verifies all services start without conflicts, tests DNS → AdGuard and HTTPS → Caddy → backend flow

### Level 3: CI (GitHub Actions)

```bash
nix build .#checks.x86_64-linux.adguard-test
nix build .#checks.x86_64-linux.caddy-test
nix build .#checks.x86_64-linux.integration
```

- Runs on every push and PR
- No secrets needed — tests use dummy/default values in isolated VMs
- Uses Cachix (free tier) for Nix build caching to keep CI fast

## User-Specific Customization

Users who clone this repo customize via:

1. `hosts/nuc/local.nix` — static IP, hostname, timezone (gitignored, created by setup script)
2. Secrets prompted during setup
3. Enabling/disabling modules by adding/removing imports in `hosts/nuc/default.nix`

## Future Expansion

The modular structure makes adding services straightforward — create a new `modules/foo.nix`, import it in `hosts/nuc/default.nix`, optionally add a test. Potential additions:

- Jellyfin (media server)
- Nextcloud (file sync)
- Grafana + Prometheus (monitoring)
- Forgejo (git hosting)
